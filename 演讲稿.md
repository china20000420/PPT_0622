`[→P3]` 

GPU 做矩阵乘法要求 batch 内所有 sequence 长度一致，几十亿篇长短不一的文档必须统一成同一个长度。要么 Padding——短文档后面填 PAD token，GPU 白算；要么 Packing——把多个短文档拼成一段等长序列，利用率 100%。目前主流的做法还是会使用后面的拼接packing的方法，这样会极大程度的利用gpu算力。

就拿右边的RefinedWeb数据集举例，平均文档长度大概是 600 tokens，八成以上不到一千，只有 5% 超过 2048。L 通常设4K——一篇文档连三分之一都填不满，必须多篇拼在一起。

所以为了利用满算力，拼接是必须的。而怎么拼、哪些文档拼一起、切不切文档，直接决定了 PPT 下面三个指标：截断率XX、噪声比例XX、幻觉率XX。今天要分享的四种方法，本质就是对这三个指标的不同取舍。

`[→P4]` 这页展示了截断的两种典型损害。左边代码场景——import 和定义在序列 A，调用在序列 B，截断后模型学到"没定义也能用"。右边知识——完整事实被切碎，模型学不到只能靠猜，那么这就会导致幻觉的出现，模型会凭空捏造事实。

下面这篇论文分析了截断位置对于模型性能的影响，理论分析表明XXX，这说明截断的位置越靠后，损失的信息会越大！



`[→P5]` 下面是讲第一种方法，随机拼接，他的思路是比较简单的。PPT 上三步——首先将数据打乱、加bos或EOS 拼成超长 token 流 、每满 L token 比如4K token 切一段。那么切完后的每个sequence严格 4K token，零 padding。最后一段不够4K token的直接丢弃！！！这样gpu利用率能达到 100%。

PPT 右边这个是截断率实测——当sequence长度设置为512时， 截断率 85%，2048时 截 60%，8192时 截 30%。这篇文章结果表明训练时候的sequence越短，每篇文章截断的概率是越大的！

那么可以看到随机拼接方法有两个重大问题：
1.在数据层面，数据被截断完整性被破坏
2.在模型层面，模型只能学会"语法通顺、事实乱猜"的预测模式，遇到填空就蒙一个像模像样的答案，那这就会导致幻觉的出现；




`[→P6]` 基于随机拼接方法，可以引出三条改进路线。算法层——保持框架，用组合优化最小化样本截断率。语义层——框架不变，改变文档顺序消灭噪声，让语义相似的文本组合在一起。范式层——放弃随机拼接的框架重新定义训练，DD。

## Best-Fit Packing

`[→P7]` 第一篇是亚马逊发布的bestfit算法。PPT 左边的这个图，上面五个文档，左边是Best-Fit 两阶段流程，假设拼接的是4K长度，那么第一步是将所有样本切割到4K及以下，以上可以看到蓝色的样本是只有蓝色的样本被截断了，待所有的样本切割至<=4K后，第二步开始拼接，拼接的思路作者转为装箱问题。然后右边是随机拼接的方法，可以清楚的看到他们最明显的区别就是截断率的不同。



`[→P8]` 具体说一下论文中的算法。
第一步是切割。刚刚提到，假设sequence是4K，那么小于等于4K的完整保留，大于4K的没办法只能被截断。论文里表示只切约 5% 的文档，其余完整保留。但随机拼接里 有 33% 的文档被截断.

第二步，BFD 装箱的思路是。首先将切割完的样本排序降序。BFD 贪心——找剩余空间最小的 bin，填小留大，并使用线段树加速，也就是利用 BFD 不区分同容量 bin 的性质。可以看到这个算法的复杂度是近似线性的，对于海量数据比较友好！！！

其实还有第三步，就是所有样本装完后会使用padding token填充！

论文中有提到，对于两个数据集，padding率为百分之0.00几，padding率几乎为0，截断率显著下降

`[→P9]` 刚刚提到了作者使用线段树优化。BFD 不区分相同剩余容量的箱子——两个都剩 5 tokens 完全等价。所以数据结构只需为 L 种容量值建，不用为 N 个箱子建。右边列出来使用线性树加速和普通方法的对比，运行时间——1B 文档 7.3h→3.0h；2B 文档 15.3h→6.2h。

`[→P10]` 再看看截断率对比图。虚线是文档数量分布，实线是截断数量，分 随机拼接和 Best-Fit，各画了 sequence是2K 和 sequence为8K。核心区别：随机拼接全区间有截断——按位置随机切。Best-Fit红色和蓝色线条对应的 的截断线在 l ≤ L 的区域直接归零——长度不超过 L 的文档一刀不切。

`[→P11]` 
PPT 上列了这篇论文的核心结果。阅读 +4.7%——测试的是模型给一篇文章后能不能真正理解内容并正确回答问题，八个数据集全部正向。NLI +9.3%——自然语言推理，测试的是模型判断两个句子之间逻辑关系的能力，是蕴含、矛盾还是中立，ANLI 这个最难的数据集上提升最大。上下文跟随 +16.8%——22 项中提升最大，测试的是模型到底会不会认真看并利用你给它的上文，还是依赖自己预训练背下来的东西，这个能力对截断最敏感所以提升最猛。程序合成 +15.0%——测试的是给一段自然语言描述的功能需求，模型能不能写出正确的代码，HumanEval pass@1 从 28.3 跳到 32.5，因为代码里变量定义和使用不再被切分，能力直接被释放。幻觉 −58.3%——测试的是代码生成中模型会不会使用从未定义过的变量名，从 3.6% 降到 1.5%，截断解决后模型不再"编造"变量。填充增加不到十万分之三，22 任务零退化，HF TRL 已集成。

`[→P12]` 定位。Best-Fit 在训练效率、时间复杂度、工程集成维度都比较优秀。局限在于只从长度优化，不区分语义。下一篇文章将从语义的角度来实现数据拼接！！

## In-Context Pretraining

`[→P13]` 这一篇是Meta发布的论文，从语义相似的角度来实现数据拼接。ICLM 换了个角度，尽可能的将语义相关的文档拼接在一个sequence中，如果同一个上下文窗口里的文档是语义相关的，前文就能给后文提供预测线索。左边的图是两步的思路，首先检索建文档图将相似的文档聚类，图遍历实现拼接。

具体来说，第一步：使用论文提出的 Contriever 编码器，将每篇文档编码为 768 维稠密向量，然后通过 FAISS 高效检索，为每篇文档找出语义最相近的 top-10 最近邻。这一步的核心是把"内容相关的文档"拉到彼此附近，为后续拼接建立语义基础。第二步：
建图：把每篇文档当作图中的一个节点，如果两篇文档互为 top-10 最近邻，就在它们之间连一条边。用贪心遍历在文档图上寻找一条能覆盖所有节点的路径，确保语义相近的文档在路径上相邻。然后将这条路径按固定长度 L（如比如如图的8K token）切成训练序列。由于语义相关的文档被排在了一起，文档边界尽量落在序列切分点，大幅减少了跨文档截断。
以上是这篇文章大致的思路



`[→P15]` Contriever 的核心是无监督对比学习——同一文档的两个随机片段作为正样本对，不同文档的片段作为负样本对，用 InfoNCE 损失优化。整个训练过程不需要任何人工标注，纯粹从文本自身的共现结构中学习语义表示。
嵌入公式如 PPT 所示：对文档每个 token 取最后一层隐藏状态，做平均池化，输出 768 维稠密向量。余弦相似度计算两文档语义接近程度，每篇文档取 top-10 最近邻，k=10  是论文消融后的最优值——k  太小图太稀疏容易孤立断裂，k  太大热门文档主导遍历，10 是连通性与语义多样性的最佳平衡点。
这里有个关键对比：论文做了 BM25 替换 Contriever 的消融实验，效果明显下降。原因在于 BM25 基于词袋重叠——只能找到"用词相同"的文档，完全抓不住"用词不同但语义相同"的文档对。比如"机器学习"和"深度学习"在 BM25 眼里可能毫无关系，但 Contriever 的密集嵌入能把它们拉到同一语义空间。


`[→可不讲]` 这页讲第二步——贪心图遍历。PPT 上给出了目标函数，在文档图 G 上找一条覆盖每个文档恰好一次的哈密顿路径，最大化相邻文档的相似度之和。
算法是双层贪心近似。外层——每次从还没访问的节点里选度数最小的，作为新链段的起点。为什么选最小度数？PPT 上右边有分析。图的度数分布是不均匀的，热门文档度数极高，冷门文档度数很低。如果最后再处理冷门文档，它的几个邻居可能已经被前面链段访问完了——它就成孤点了，只能零权边跳转到下一个链段，路径的整体连贯性变差。反过来，优先处理这些"难安排"的文档，趁着邻居还没被访问，可以在它们之间形成短但连贯的链段。这就是最小度数优先启发式——Min-Degree-First。内层——从当前节点出发，在未访问邻居里选余弦相似度最高的作为下一步，一直走到当前连通分量的所有节点都访问过了，回外层选新起点。



`[→P15]` k 值是 ICLM 最关键的参数，直接控制文档图的连接密度。k=1 的时候图太稀疏，大量孤立节点，路径走着走着就断了；k=5 有所改善，但还是会经常遇到断裂；k=10 是论文消融确定的最优值，连通性和语义多样性取得了最佳平衡；k=20 就过头了，图过于稠密，少数热门文档被过度连接，遍历路径被它们主导，反而丧失了多样性。
另外，语义去重是 ICLM 的一个附带收益。近重复文档对在检索中相似度常常超过 0.98，会在图中形成密集子图，构建过程中可以直接识别并移除。论文做了消融实验，确认不去重效果会显著退化，因为这些近重复文档会扭曲图的度数分布和遍历路径。



`[→P16]` 这页是本篇论文的实验结果！第一行，In-Context Learning，+8%。测的是模型能不能靠上下文里的几个示例快速适应新任务，不靠微调。ICLM 预训练时就在多篇相关文档里学，跟这个评估格式天然对齐。
第二行，阅读理解，四种方法里最高。给一篇文章让模型真正读懂并回答问题。ICLM 的输入本来就是多篇相关文档拼在一起，练的就是"综合信息做判断"。
第三行，忠实度，+16%。测的是上下文信息和模型自己记忆冲突时，模型信哪个。ICLM 更倾向于信上下文，而不是凭记忆瞎猜。
第四行，检索增强，+9%。给外部知识库，测模型能不能从中找到正确信息来回答问题，考的是"检索+利用"的联合能力。
第五行，长上下文推理，+5%。测的是在长上下文中保持注意力、定位关键信息的能力。
第六行，困惑度，全部更低。这是基础语言建模能力，Wikipedia、Arxiv、Books 三个领域，0.3B 到 7B 全规模，ICLM 都优于 Standard。

可以看到ICLM 的全面性非常好，六个维度全部正向提升，并且论文提到它的优势随数据规模单调递增——从 100M 到 10B tokens，优势从 +4% 涨到 +11%，没有饱和。

`[→P17]` 这页把三种方法放在一起对比，我们按表中维度逐行过一遍。
操作方式。随机拼接最简单，打乱文档顺序后直接拼起来切段。Best-Fit 在此基础上多了两步——先 Segmentation 把长文档切成 chunk，再装箱优化减少截断。ICLM 在 Best-Fit 基础上再多一步，用 Contriever 做语义检索和图遍历，把相关文档排在一起。
优化目标。三种方法盯的是三个东西——随机拼接盯 GPU 利用率，只要序列长度固定、零 padding，算力就不浪费；Best-Fit 盯 文档完整性，让每篇文档尽量不被切碎；ICLM 盯 语义连贯性，让相邻文档内容相关，模型注意力有东西可学。从"不浪费算力"到"不浪费数据"到"不浪费注意力"，逐层深入。
截断率。随机拼接约 60%，六成文档被切碎；Best-Fit 降到 5% 以下，装箱优化效果显著；ICLM 进一步到 1% 以下，语义排序让文档边界更对齐切分点。每换一个方法，截断率降一个数量级。
跨文档注意力。随机拼接里，相邻文档八竿子打不着，注意力是纯粹噪声；Best-Fit 里噪声还在，但至少文档完整了，不会把一篇论文拦腰斩断；ICLM 里相邻文档语义相关，注意力从噪声变成了可能有用的信号——模型真的在"读相关的东西"。
